## 🤖 2.2 Secure cookies + session TTL/rotation

**Что сделать в коде (AI):**
- **NestJS Auth:** API выпускает сессионные cookies с `HttpOnly`, `Secure` и подходящим `SameSite`; TTL и ротация задаются в backend-конфигурации. Next.js не создает и не валидирует авторитетную сессию самостоятельно.

**Проверка:** В DevTools → Application → Cookies у сессионной куки стоят HttpOnly, Secure, SameSite. JS не может прочитать значение.

---

## 🤖 2.3 CSRF protection

**Что сделать в коде (AI):**
- Для cookie-based auth и любых мутирующих POST: проверять `Origin`/`Referer` на сервере (allowlist своих доменов) или использовать CSRF-токен (например в формах и API).
- CSRF-защита реализуется централизованно в NestJS API для всех cookie-authenticated mutations.

**Проверка:** POST с другого origin (или без валидного токена) возвращает 403 / invalid token.

---

## 🤖 2.4 RBAC server-side

**Что сделать в коде (AI):**
- В каждом защищенном NestJS endpoint после аутентификации проверять роль, компанию и права через guard/policy. Не полагаться на скрытие кнопок или redirects в Next.js.
- Централизовать проверки в NestJS guards/policies, чтобы новый controller не мог обойти правила.

**Проверка:** `curl` без токена или с токеном без прав на админ-действие возвращает 401/403, даже если в UI кнопка есть.

---

## 🤖 2.7 Logout invalidates session/token

**Что сделать в коде (AI):**
- NestJS logout endpoint инвалидирует backend-сессию/refresh token и очищает cookie. После выхода повторный запрос с тем же токеном/кукой должен получать 401.

**Проверка:** После logout повторный запрос с тем же cookie/token к защищённому API → 401 Unauthorized.

## 👤 2.1 ADR: схема NestJS auth

**Что сделать вручную (документ):**
- Создать короткий ADR или раздел в Docs: cookie/session или access/refresh стратегия, где NestJS проверяет сессию, как устроены роли и ownership, где хранятся refresh/session records и как работает logout/revocation.

**Проверка:** Любой разработчик за 5 минут понимает из документа, как устроена auth.

## 👤 2.5 Admin MFA mandatory

**Куда зайти → что сделать:**
- Для backend-owned auth реализовать или подключить MFA-провайдер; для P0 админов минимум — обязательная 2FA (например TOTP) и NestJS policy, запрещающая административный доступ без пройденной MFA.

**Проверка:** Админ не может войти без прохождения MFA.

## 👤 2.6 Password policy (если не только OAuth)

**Что сделать вручную (Docs/Policy):**
- **Clerk:** **Configure → Email, Phone, Username** / настройки пароля: минимальная длина, сложность (если есть опции). Зафиксировать в политике/документе.
- **Свой логин:** в коде или в настройках провайдера задать минимальную длину и при необходимости проверку сложности; описать в документации.

**Проверка:** В настройках провайдера или в коде задана политика; в доке она описана.
